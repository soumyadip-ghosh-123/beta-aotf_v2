# Migrate Legacy Users

This guide walks through the production migration from the legacy **Academy of Tutorials** database into AOTF v2 (Clerk + new MongoDB). It matches the workflow used when importing ~550+ paid users.

## Prerequisites

- Production Clerk instance (`pk_live_` / `sk_live_`) with Google OAuth **enabled** for migrated users
- Read access to legacy MongoDB (`MONGODB_URI_LEGACY`)
- Write access to new app MongoDB (`MONGODB_URI`)
- Clerk webhook configured (backup path for Mongo seeding)

## Step 1 — Configure Environment

In `.env.local` (or CI secrets for a one-off run):

```bash
NODE_ENV=production

# Target Clerk (production)
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_live_...
CLERK_SECRET_KEY=sk_live_...

# Databases
MONGODB_URI_LEGACY=mongodb+srv://...@cluster/...   # legacy cluster
MONGODB_URI=mongodb+srv://...@cluster/test         # new app DB
```

`NODE_ENV=production` ensures migration scripts use public DNS (`1.1.1.1`) for Atlas SRV resolution on machines with broken local DNS.

## Step 2 — Dry Run

```bash
cd scripts
node migrate-teachers-to-clerk.mjs
node migrate-freelancers-to-clerk.mjs
```

Review the summary counts. Expect one line per paid user with email, name, role, and legacy ID. No Clerk or Mongo writes occur without `--live`.

## Step 3 — Live Migration

```bash
node migrate-teachers-to-clerk.mjs --live
node migrate-freelancers-to-clerk.mjs --live
```

Each script:

1. Connects to legacy DB `academy-of-tutorials-freelancers`
2. Loads records with `registrationFeeStatus: "paid"`
3. Creates Clerk users (or skips if email exists)
4. Seeds `User`, `Profile`, and `OnboardingDetails` in the new MongoDB

Allow several minutes for large batches (rate-limited ~250 ms between Clerk API calls).

## Step 4 — MongoDB Backfill (if needed)

If the summary shows Clerk creates but Mongo seed failures, or users are missing from `/admin/users`:

```bash
node sync-clerk-users-to-mongodb.mjs --live
```

## Step 5 — Verify

| Check | How |
|---|---|
| Clerk dashboard | User count matches paid legacy records |
| `/admin/users` | Teachers and candidates tabs populated |
| Sample user metadata | `migratedFromLegacy: true`, `registrationFeeStatus: "paid"` |
| Sign-in | Google with same Gmail links to pre-created account |

## Step 6 — User Activation

Migrated users are **not** fully onboarded until they complete the in-app flow:

1. Sign in with **Continue with Google** (or set password via Forgot Password)
2. Complete the 3-step `/onboarding` form
3. Click **Activate Account** (skips Razorpay — already paid in legacy system)
4. `POST /api/v1/payments/activate-legacy` sets `onboardingCompleted: true`

See [Provider Onboarding](/docs/tutorials/provider-onboarding#legacy-migrated-users) and [Account Linking](/docs/explanations/account-linking).

## Troubleshooting

### `legal_accepted_at` is required

Production Clerk has **Require express consent** enabled. Migration scripts pass `skipLegalChecks: true` and `legalAcceptedAt` from legacy `termsAgreedAt`. If you see this error, ensure you are on the latest script version.

### `querySrv ECONNREFUSED`

Atlas SRV DNS failure. Confirm `NODE_ENV=production` in `.env.local` and re-run.

### `E11000 duplicate key` on `onboardingdetails`

Usually means a partial seed from an earlier run. Re-run `sync-clerk-users-to-mongodb.mjs --live` — the seed helper upserts by `clerkId` or `userId`.

### Username or email validation failures

Fix manually in Clerk for edge cases (very long email local parts, invalid emails). Example: set username `aryabandyopadhyay` for `aryabandyopadhyay.official@gmail.com`, then run sync.
