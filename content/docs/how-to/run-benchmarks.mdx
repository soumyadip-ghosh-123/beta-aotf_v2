# Run Performance Benchmarks

AOTF ships with two benchmark scripts: one for API response times and one for MongoDB query timing. Both output formatted tables and save JSON results.

## API Benchmark

Hits every documented endpoint and records median, min, and max response times.

### Setup

1. Start the dev server (or target production):
   ```bash
   pnpm dev
   ```

2. Get a Clerk token for authenticated routes. In the browser console on a logged-in page:
   ```js
   // In browser console on aotf.in (or localhost:3000)
   copy(await window.Clerk.session.getToken())
   ```

3. Get an admin token similarly while logged in as an admin.

### Run

```bash
# Against localhost (default)
pnpm benchmark:api

# Against production
BASE_URL=https://aotf.in AUTH_TOKEN=<token> ADMIN_TOKEN=<admin_token> pnpm benchmark:api

# Run 5 times per endpoint (default: 3)
RUNS=5 pnpm benchmark:api

# Suppress per-request logs (cleaner output)
QUIET=1 pnpm benchmark:api
```

### Environment Variables

| Variable | Default | Description |
|---|---|---|
| `BASE_URL` | `http://localhost:3000` | Target server URL |
| `AUTH_TOKEN` | _(empty)_ | Clerk JWT for provider routes |
| `ADMIN_TOKEN` | _(empty)_ | Clerk JWT for admin routes |
| `RUNS` | `3` | Requests per endpoint (reports median) |
| `OUTPUT` | `scripts/benchmark-results.json` | JSON output file path |
| `QUIET` | _(not set)_ | Set to `"1"` to suppress request logs |

### Interpreting Results

| Response Time | Assessment |
|---|---|
| < 100ms | Excellent (cached or simple query) |
| 100–300ms | Good (typical DB query) |
| 300–600ms | Acceptable (complex query or cold DB) |
| 600ms–1s | Slow — investigate query indexes |
| > 1s | Cold start or unoptimised query |

## Database Query Benchmark

Times representative MongoDB queries on every collection, differentiating cold-start (first call) from warm (cached) performance.

### Setup

Requires `MONGODB_URI` in `.env.local`. No dev server needed.

### Run

```bash
pnpm benchmark:db
```

### Sample Output

```
⏳ Cold start connect…
   Connected in 4821ms    ← Atlas M0 waking from auto-pause

  users / findOne (clerkId)… cold=87ms  warm=12ms
  users / countDocuments… cold=45ms  warm=8ms
  applications / find by status (limit 10)… cold=134ms  warm=23ms
  ...

┌────────────────────┬───────────────────────────────────┬──────────┬──────────┬────────────┐
│Collection          │Operation                          │Cold (ms) │Warm (ms) │Δ (ms)      │
├────────────────────┼───────────────────────────────────┼──────────┼──────────┼────────────┤
│users               │findOne (clerkId)                  │87        │12        │-75         │
│users               │countDocuments                     │45        │8         │-37         │
│applications        │find by status (limit 10)          │134       │23        │-111        │
│calendar_events     │find by date range                 │198       │34        │-164        │
└────────────────────┴───────────────────────────────────┴──────────┴──────────┴────────────┘

   DB connect: 4821ms
   Avg cold query: 118ms  |  Avg warm query: 19ms
   Warm queries are ~84% faster than cold
```

### Key Observations

- **Connect time**: On a paused M0 cluster, the initial connection takes **4–10 seconds**. The `instrumentation.ts` file pre-warms this at server startup.
- **Cold vs warm delta**: Warm queries are significantly faster because MongoDB caches query plans and WiredTiger buffers data in memory.
- **Index effectiveness**: If cold and warm times are similar and both slow, the query is likely doing a collection scan — add an index.

## Using Postman

The project includes a pre-built Postman collection at `public/docs/AOTF.postman_collection.json`.

### Import

1. Open Postman
2. **File → Import** → select `public/docs/AOTF.postman_collection.json`
3. The collection appears with all endpoints grouped by tag

### Configure Environment

Create a Postman environment with these variables:

| Variable | Value |
|---|---|
| `base_url` | `http://localhost:3000` or `https://aotf.in` |
| `clerk_token` | Your provider Clerk JWT |
| `admin_token` | Your admin Clerk JWT |

### CSRF Header

All mutation endpoints require the `Origin` header. The collection includes a pre-request script that sets it automatically:

```js
// Pre-request script (already in collection)
pm.request.headers.add({
  key: "Origin",
  value: pm.variables.get("base_url")
});
```

### Running the Full Collection

Use **Collection Runner** to run all requests in sequence and export a performance report.
