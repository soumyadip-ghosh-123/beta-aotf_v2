## Architecture

The service layer sits between route handlers and Mongoose models, encapsulating all business logic:

```
Route Handler → Service → Model → Database
```

Services are located in `lib/services/` and follow a consistent pattern.

## Service Pattern

A typical service function follows this structure:

```typescript
// lib/services/post.service.ts

export async function createPost(data: CreatePostInput, adminClerkId: string) {
  await dbConnect();

  // 1. Generate unique ID
  const postId = generatePostId();

  // 2. Create document
  const post = await Post.create({
    ...data,
    postId,
    createdByAdminClerkId: adminClerkId,
    status: "open",
  });

  // 3. Side effects (calendar, sheets, etc.) happen via model hooks

  return post;
}
```

### Key Principles

1. **Always call `dbConnect()`** before any database operation
2. **Never expose Mongoose internals** — return plain objects or mapped DTOs
3. **Side effects via hooks** — Calendar and Sheets syncing happens through Mongoose post-hooks on the model
4. **Throw `AppError` subclasses** for operational errors — the route handler catches them

## API Utilities

The `lib/api-utils.ts` module provides reusable helpers for route handlers:

### CSRF Protection

```typescript
import { checkCsrfOrigin } from "@/lib/api-utils";

export async function POST(request: NextRequest) {
  const csrfError = checkCsrfOrigin(request);
  if (csrfError) return csrfError;

  // ... handle request
}
```

Validates that the `Origin` or `Referer` header matches the application host.

### Rate Limiting

```typescript
import { checkRateLimit } from "@/lib/api-utils";
import { createRateLimiter } from "@/lib/rate-limit";

const limiter = createRateLimiter({ windowMs: 60_000, max: 5 });

export async function POST(request: NextRequest) {
  const rateLimitError = checkRateLimit(limiter, getClientIp(request));
  if (rateLimitError) return rateLimitError;

  // ... handle request
}
```

The rate limiter uses an in-memory sliding window. For multi-instance deployments, swap to Redis (e.g., `@upstash/ratelimit`).

### Content-Type Validation

```typescript
const ctError = checkJsonContentType(request);
if (ctError) return ctError;
```

### Centralised Error Handling

```typescript
import { handleApiError } from "@/lib/api-utils";

export async function POST(request: NextRequest) {
  try {
    // ... handle request
  } catch (error) {
    return handleApiError(error, "POST /api/v1/posts");
  }
}
```

`handleApiError` maps errors to appropriate HTTP responses:

| Error Type | Status | Response |
|-----------|--------|----------|
| Zod validation | `400` | Field-level errors |
| `ValidationError` | `400` | Custom message + field errors |
| `NotFoundError` | `404` | Resource not found |
| `ConflictError` | `409` | Duplicate entry |
| `AppError` | Varies | Custom status code |
| Mongoose duplicate key | `409` | Generic duplicate message |
| `SyntaxError` | `400` | Invalid JSON body |
| Unknown | `500` | "Internal server error" (no leak) |

## Custom Error Classes

Defined in `lib/errors.ts`:

```typescript
class AppError extends Error        // Base: any status code
class ValidationError extends AppError  // 400: with fieldErrors
class NotFoundError extends AppError    // 404: resource not found
class ConflictError extends AppError    // 409: duplicate/conflict
class InternalError extends AppError    // 500: unexpected
```

All errors carry an `isOperational` flag. `InternalError` sets it to `false` to distinguish unexpected crashes from controlled failures.
