# User, Profile & OnboardingDetails Models

## User

**Collection**: `users`  
**Source**: [`lib/models/User.ts`](file:///d:/aotf_v2/lib/models/User.ts)

The `User` model is the **primary account record** for all providers. It's created by the Clerk webhook on `user.created` and is the source of truth for account status.

### Schema

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `_id` | `ObjectId` | Auto | — | MongoDB document ID |
| `clerkId` | `string` | ✅ | — | Clerk user ID (indexed, unique) |
| `username` | `string` | ✅ | — | Lowercase, unique (case-insensitive index) |
| `role` | `enum` | ✅ | `"teacher"` | `teacher`, `teacher_candidate`, `admin` |
| `plan.current` | `enum` | ✅ | `"teacher"` | `teacher`, `teacher_candidate` |
| `plan.hasTuitionAccess` | `boolean` | — | `false` | Can apply to tuition posts |
| `plan.hasCandidateAccess` | `boolean` | — | `false` | Can apply to job/project posts |
| `plan.activatedAt` | `Date` | — | `null` | When plan was activated (payment time) |
| `onboardingCompleted` | `boolean` | — | `false` | Gates access to the platform |
| `status` | `enum` | — | `"active"` | `active`, `blocked`, `deleted` |
| `registrationPaymentId` | `ObjectId` | — | `null` | Ref to `Payment` document |
| `deletionWarningEmailSentAt` | `Date` | — | `null` | When the 5-day warning email was sent |
| `createdAt` | `Date` | Auto | — | From `timestamps: true` |
| `updatedAt` | `Date` | Auto | — | From `timestamps: true` |

### Indexes

```ts
{ clerkId: 1 }     // unique
{ username: 1 }    // unique, case-insensitive collation
{ status: 1 }
```

### Notes

- `username` is **immutable** after creation — the Clerk webhook logs and ignores username changes
- `plan.hasTuitionAccess` and `hasCandidateAccess` are set by `/api/v1/payments/verify` after payment
- `status: "deleted"` is a soft delete — the Clerk account still exists but DB checks block access

## OnboardingDetails

**Collection**: `onboarding_details`  
**Source**: `lib/models/OnboardingDetails.ts`

Tracks the onboarding state machine. Includes a **TTL index** for auto-deletion of incomplete onboarding records after 30 days.

### Schema

| Field | Type | Default | Description |
|---|---|---|---|
| `userId` | `ObjectId` | — | Ref: `User._id` |
| `clerkId` | `string` | — | Clerk user ID |
| `phone` | `string` | `null` | Contact phone |
| `whatsapp` | `string` | `null` | WhatsApp number |
| `address` | `string` | `null` | Address |
| `teachingExp` | `string` | `null` | Teaching experience range |
| `jobExp` | `string` | `null` | Work experience range |
| `qualification` | `string` | `null` | Highest degree |
| `board` | `string` | `null` | Board preference |
| `plan` | `string` | `null` | Selected plan |
| `status` | `enum` | `"incomplete"` | `incomplete`, `complete` |
| `expiresAt` | `Date` | `now + 30 days` | TTL — set to `null` for paid users |

### TTL Index

```ts
OnboardingDetailsSchema.index(
  { expiresAt: 1 },
  { expireAfterSeconds: 0 } // expires at the expiresAt datetime
);
```

**For paid users**: `expiresAt` is set to `null` immediately after payment — the TTL never fires.

**For non-paying users**: After 30 days, the document is automatically deleted by MongoDB. A cron job (or Vercel cron) also sends a 5-day warning email by setting `User.deletionWarningEmailSentAt`.
