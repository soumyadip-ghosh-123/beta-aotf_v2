# Webhooks

AOTF has two webhook receivers: one for **Clerk** (user lifecycle events) and one for **Razorpay** (payment events).

## Clerk Webhook

**Endpoint**: `POST /api/v1/webhooks/clerk`  
**Source file**: [`app/api/v1/webhooks/clerk/route.ts`](file:///d:/aotf_v2/app/api/v1/webhooks/clerk/route.ts)

### Authentication

Clerk uses [Svix](https://www.svix.com/) to sign webhook deliveries. Every request includes:

```http
svix-id: msg_xxx
svix-timestamp: 1700000000
svix-signature: v1,xxx
```

The handler verifies with `@clerk/nextjs/webhooks`:
```ts
import { verifyWebhook } from "@clerk/nextjs/webhooks";

evt = await verifyWebhook(req, {
  signingSecret: process.env.CLERK_WEBHOOK_SECRET!,
});
```

If verification fails → **HTTP 400** (Clerk will retry).

### Why Webhooks Bypass proxy.ts

`clerkMiddleware` strips Svix headers from requests. `proxy.ts` explicitly passes webhook paths through **before** the middleware runs:

```ts
// proxy.ts
if (pathname.startsWith("/api/v1/webhooks/")) {
  return NextResponse.next({ request: { headers: requestHeaders } });
}
```

### Idempotency Guard

Before processing any event, the handler inserts a record into the `WebhookEvent` collection:

```ts
await WebhookEvent.create({
  provider: "clerk",
  event: eventType,
  entityId: data.id,
  payload: data,
  signature: req.headers.get("svix-signature"),
});
```

If this insert fails with error code **11000** (duplicate key), the event was already processed — the handler returns **HTTP 200** immediately without re-processing.

**On processing failure**: The `WebhookEvent` document is **deleted** so that Clerk's automatic retry can attempt the event again on the next delivery.

### Handled Events

#### `user.created`

Fires when a new user registers (self-service sign-up) or is created via migration script / admin API.

| Step | Action |
|---|---|
| 1 | Check `publicMetadata.isAdmin` — route to admin or user handler |
| 2 | **User handler**: Create or update `User` + `Profile` (match by `clerkId` or `username`) |
| 3 | If `migratedFromLegacy: true` and `registrationFeeStatus: "paid"`: upsert `OnboardingDetails` with `expiresAt: null` |
| 4 | Update Clerk `publicMetadata` — preserve migration fields (`legacyPlan`, `legacyTeacherId`, `registrationFeeStatus`) |
| 5 | Set `onboardingCompleted: false` until payment verify or `activate-legacy` |
| 6 | Retry up to 3 times on transient 404 (Clerk propagation delay) |

**Migration scripts** call `seedClerkUserInMongo()` immediately after Clerk user creation. The webhook is a **backup** if the script seed fails or the user signs in before seeding completes. See [Seed Teachers & Candidates](/docs/how-to/seed-users).

#### `user.updated`

Fires when a user's Clerk profile changes.

| Condition | Action |
|---|---|
| Admin account | Sync `username` and `name` to `Admin` document |
| Regular user | Sync `avatarUrl` to `Profile`; username changes are **locked** (ignored) |

#### `user.deleted`

Fires when a Clerk user is deleted.

| Account type | Action |
|---|---|
| Admin | Soft delete: `Admin.isActive: false`, `terminatedAt: now` |
| Provider | Soft delete: `User.status: "deleted"`, `Profile.isPublic: false`, delete `OnboardingDetails` |

**Note**: MongoDB data is never hard-deleted on user deletion. This preserves audit trails.

### Event Table

| Event | Creates | Updates | Deletes |
|---|---|---|---|
| `user.created` (provider) | `User`, `Profile` | Clerk metadata | — |
| `user.created` (admin) | `Admin` | Clerk metadata | — |
| `user.updated` | — | `Admin` or `Profile` | — |
| `user.deleted` (provider) | — | `User`, `Profile` | `OnboardingDetails` |
| `user.deleted` (admin) | — | `Admin` (soft) | — |

## Razorpay Webhook

**Endpoint**: `POST /api/v1/webhooks/razorpay`  
**Source file**: [`app/api/v1/webhooks/razorpay/route.ts`](file:///d:/aotf_v2/app/api/v1/webhooks/razorpay/route.ts)

### Authentication

Razorpay signs each request with `X-Razorpay-Signature`:

```ts
const expectedSignature = crypto
  .createHmac("sha256", process.env.RAZORPAY_WEBHOOK_SECRET!)
  .update(JSON.stringify(body))
  .digest("hex");

if (expectedSignature !== req.headers.get("x-razorpay-signature")) {
  return 400; // Rejected
}
```

### Handled Events

| Event | Action |
|---|---|
| `payment.captured` | Set `Payment.status: "captured"`, ensure onboarding is marked complete |
| `payment.failed` | Set `Payment.status: "failed"`, log failure reason |

### Relationship to /payments/verify

Both the client-side `/api/v1/payments/verify` and the webhook confirm the same payment. They are intentionally redundant:

- `/payments/verify` → Fast, client-triggered, immediate UX feedback
- Webhook → Server-to-server, reliable, reconciliation safety net

If the client verify fails (e.g., browser closed), the webhook still marks the payment correctly.
