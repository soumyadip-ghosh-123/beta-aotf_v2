# API Overview

All AOTF backend routes live under `app/api/`. This page explains the conventions, security layers, and patterns used across every route.

## Base URL & Versioning

```
Production:   https://aotf.in/api/v1/...
Development:  http://localhost:3000/api/v1/...
```

All application endpoints are under `/api/v1/`. Infrastructure endpoints (`/api/health`, `/api/openapi.json`, `/api/search`) are unversioned.

## Authentication

AOTF uses **Clerk** for authentication. Every protected route reads the session from the Clerk JWT.

### How Auth Works in Route Handlers

```ts
// Pattern used in every protected route
import { auth } from "@clerk/nextjs/server";

export async function GET(req: NextRequest) {
  const { userId, sessionClaims } = await auth();

  if (!userId) {
    return NextResponse.json({ error: "Authentication required" }, { status: 401 });
  }

  // sessionClaims.publicMetadata contains role, permissions, onboardingCompleted
  const { role, isAdmin, permissions } = sessionClaims.publicMetadata as AdminMetadata;
  // ...
}
```

### Auth Flow Summary

1. **Client sends request** with `Authorization: Bearer <clerk_jwt>` header
2. **`proxy.ts`** (Clerk middleware) validates the JWT and attaches `auth()` to the request context
3. **Route handler** calls `await auth()` to get `userId` and `sessionClaims`
4. **`sessionClaims.publicMetadata`** contains role, permissions, and `onboardingCompleted` — no extra DB call needed for basic auth checks

> **Note for Admin routes**: JWT claims are checked first for performance, but the DB is always consulted for permission verification on admin routes. The `Admin` collection is the source of truth.

## CSRF Protection

All **mutation endpoints** (POST, PATCH, PUT, DELETE) that are not webhook handlers require the `Origin` or `Referer` header to match the application host.

### Implementation

```ts
// lib/api-utils.ts
export function checkCsrfOrigin(req: NextRequest): boolean {
  const origin = req.headers.get("origin") ?? req.headers.get("referer") ?? "";
  const appUrl = process.env.NEXT_PUBLIC_APP_URL ?? "";
  return origin.startsWith(appUrl);
}
```

### Using It in a Route

```ts
export async function POST(req: NextRequest) {
  if (!checkCsrfOrigin(req)) {
    return NextResponse.json({ error: "CSRF check failed" }, { status: 403 });
  }
  // ...
}
```

### Webhook Exemption

Webhook endpoints (`/api/v1/webhooks/*`) are **exempt** from CSRF checks — they come from external services (Clerk, Razorpay) that send their own signature headers instead.

## Rate Limiting

Sensitive public endpoints use an **in-memory sliding-window rate limiter** defined in `lib/rate-limit.ts`.

```ts
// lib/rate-limit.ts
const limiter = createRateLimiter({ windowMs: 60_000, max: 5 });
const result = limiter.check(ip);

if (!result.allowed) {
  return NextResponse.json(
    { error: "Too many requests. Please try again later." },
    {
      status: 429,
      headers: {
        "Retry-After": String(Math.ceil((result.resetAt - Date.now()) / 1000)),
        "X-RateLimit-Remaining": "0",
        "X-RateLimit-Reset": String(result.resetAt),
      },
    }
  );
}
```

> **Limitation**: This rate limiter is **in-memory and single-instance**. On Vercel's multi-instance serverless environment, each function instance has its own counter. For true distributed rate limiting, this should be upgraded to **Upstash Redis + @upstash/ratelimit**. See [Areas of Improvement](/docs/explanations/improvements).

### Currently Rate-Limited Routes

| Route | Window | Max |
|---|---|---|
| `POST /api/v1/enquiry` (public) | 60 seconds | 5 requests |
| `POST /api/admin/join` | 60 seconds | 5 requests |

## Error Handling

All routes use a centralised `handleApiError` utility:

```ts
// lib/api-utils.ts
export function handleApiError(
  error: unknown,
  context?: string,  // route label, e.g. "GET /api/v1/admin/posts"
): NextResponse {
  // Zod validation → 400 with fieldErrors
  // AppError subclasses → their statusCode (4xx operational errors are NOT sent to Sentry)
  // Mongoose duplicate key → 409
  // SyntaxError (bad JSON) → 400
  // Catch-all → 500
  reportError(error, { route: context });  // only when shouldReportToSentry() passes
  return NextResponse.json({ error: "Internal server error" }, { status: 500 });
}
```

Unexpected 500s are reported to Sentry via `reportError()` in `lib/sentry-report.ts`. Validation failures, operational 4xx errors, and duplicate keys are handled locally and not reported. See [Monitoring with Sentry](/docs/explanations/monitoring-sentry) for the full filtering rules and coverage map.

### Standard Error Response Shape

```json
{
  "error": "Human-readable message",
  "fieldErrors": {
    "fieldName": ["Error message 1", "Error message 2"]
  }
}
```

`fieldErrors` is only present for validation failures.

## Standard HTTP Status Codes

| Status | Meaning | When Used |
|---|---|---|
| `200` | OK | Successful GET, PATCH, DELETE |
| `201` | Created | Successful POST (resource created) |
| `400` | Bad Request | Validation error, bad ObjectId |
| `401` | Unauthorized | No Clerk session |
| `403` | Forbidden | Insufficient permissions or CSRF fail |
| `404` | Not Found | Resource doesn't exist |
| `409` | Conflict | Duplicate key (e.g., already applied) |
| `429` | Too Many Requests | Rate limit exceeded |
| `500` | Internal Server Error | Uncaught exception |
| `503` | Service Unavailable | DB unreachable (health check only) |

## How to Read an API Route File

Every API route file follows this structure:

```ts
// app/api/v1/posts/route.ts

import { NextRequest, NextResponse } from "next/server";
import { auth } from "@clerk/nextjs/server";
import dbConnect from "@/lib/db";                    // 1. Always connect first
import Post from "@/lib/models/Post";
import { checkCsrfOrigin, handleApiError } from "@/lib/api-utils";

export async function GET(req: NextRequest) {
  try {
    await dbConnect();                               // 2. Connect to MongoDB
    // Optional: const { userId } = await auth();   // 3. Auth check (if protected)
    // Optional: if (!checkCsrfOrigin(req)) ...     // 4. CSRF check (mutations only)

    // 5. Parse params / body
    const { searchParams } = req.nextUrl;
    const page = parseInt(searchParams.get("page") ?? "1");

    // 6. Query MongoDB
    const posts = await Post.find({ status: "open" })
      .skip((page - 1) * 20)
      .limit(20)
      .lean();

    // 7. Return response
    return NextResponse.json({ posts, page });
  } catch (error) {
    return handleApiError(error);                    // 8. Centralised error handling
  }
}

export async function POST(req: NextRequest) {
  try {
    await dbConnect();
    const { userId, sessionClaims } = await auth(); // Auth required
    if (!userId) return NextResponse.json({ error: "Authentication required" }, { status: 401 });

    if (!checkCsrfOrigin(req)) {                    // CSRF required for mutations
      return NextResponse.json({ error: "CSRF check failed" }, { status: 403 });
    }

    const body = await req.json();
    // ... validate, create, return
  } catch (error) {
    return handleApiError(error);
  }
}
```

## Helmet-Style Security Headers

AOTF applies security headers at the Next.js config level:

```js
// next.config.ts — security headers applied to all routes
{
  "Content-Security-Policy": "default-src 'self'; ...",
  "X-Frame-Options": "DENY",
  "X-Content-Type-Options": "nosniff",
  "Referrer-Policy": "strict-origin-when-cross-origin",
  "Permissions-Policy": "camera=(), microphone=(), geolocation=()"
}
```

These are applied globally — individual routes don't need to set them.
