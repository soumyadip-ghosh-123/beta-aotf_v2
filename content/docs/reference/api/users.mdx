# Users, Profile & Onboarding APIs

## Admin User Management

Provider account listing and management for admins is handled by the **Admin App Users API**, not `/api/v1/users`.

See [Admin App Users API](/docs/reference/api/app-users) and [Manage Users in Admin](/docs/how-to/admin-manage-users).

The public user endpoint is profile lookup by username only:

```http
GET /api/v1/users/:username
```

No authentication required for public profile data (subject to `isPublic` on the profile).

---

## Current User (Me)

### Get Current User

```http
GET /api/v1/me
Authorization: Bearer <clerk_jwt>
```

Returns the current user's document + profile. Available to any authenticated user.

**Response 200**:
```json
{
  "user": {
    "_id": "...",
    "clerkId": "user_...",
    "username": "teacher1",
    "role": "teacher",
    "plan": { "hasTuitionAccess": true, "hasCandidateAccess": false },
    "onboardingCompleted": true,
    "status": "active"
  },
  "profile": {
    "displayName": "John Doe",
    "bio": "Experienced Math tutor",
    "subjects": ["Mathematics", "Physics"],
    "experience": 5,
    "avatarUrl": "https://res.cloudinary.com/..."
  }
}
```

## Profile

### Get Own Profile

```http
GET /api/v1/profile
Authorization: Bearer <clerk_jwt>
```

Returns the complete profile for the currently authenticated user.

---

### Get Public Profile by Username

```http
GET /api/v1/profile/:username
```

No authentication required. Returns profile data for the provider's public-facing page at `/u/[username]`.

**Response 404** if profile `isPublic: false` or username not found.

---

### Update Own Profile

```http
PATCH /api/v1/profile
Authorization: Bearer <clerk_jwt>
Content-Type: application/json
Origin: https://aotf.in
```

**Request Body** (all fields optional):
```json
{
  "displayName": "John Doe",
  "bio": "Experienced Math tutor",
  "location": "Kolkata, WB",
  "websiteUrl": "https://johndoe.com",
  "subjects": ["Mathematics", "Physics"],
  "experience": 5,
  "gender": "male",
  "qualification": "B.Sc. Mathematics",
  "board": "CBSE",
  "phone": "9876543210",
  "whatsapp": "9876543210",
  "address": "123, Park Street, Kolkata",
  "teachingExp": "2-5",
  "jobExp": "0-1",
  "socialLinks": {
    "linkedin": "https://linkedin.com/in/johndoe",
    "twitter": "https://twitter.com/johndoe",
    "github": "https://github.com/johndoe"
  }
}
```

**Validation**:
- `bio`: Max 300 characters
- `address`: Max 200 characters
- `gender`: `"male"` | `"female"` | `"other"`
- `board`: `"CBSE"` | `"ICSE"` | `"WB"`
- `teachingExp` / `jobExp`: `"0-1"` | `"2-5"` | `"6-10"` | `"10+"`

## Onboarding

### Get Onboarding State

```http
GET /api/v1/onboarding
Authorization: Bearer <clerk_jwt>
```

Returns current onboarding progress, saved step data, and flags such as `paymentPaidButNotOnboarded` (paid via Razorpay but onboarding not marked complete — contact support).

### Save Onboarding Step

```http
PATCH /api/v1/onboarding
Authorization: Bearer <clerk_jwt>
Content-Type: application/json
Origin: https://aotf.in
```

Saves personal/professional fields between steps. Refreshes `OnboardingDetails.expiresAt` (+72 hours from save).

**Request Body** (fields optional):
```json
{
  "phone": "9876543210",
  "whatsapp": "9876543210",
  "address": "123, Park Street, Kolkata",
  "teachingExp": "2-5",
  "qualification": "B.Sc."
}
```

### Complete Onboarding (Payment)

Onboarding completion is **not** a single `POST /api/v1/onboarding` call. It happens via:

| Path | Endpoint |
|---|---|
| New users (Razorpay) | `POST /api/v1/payments/verify` after successful payment |
| Migrated legacy users | `POST /api/v1/payments/activate-legacy` after **Activate Account** |

See [Payments API](/docs/reference/api/payments).

**State machine on completion**:
- `OnboardingDetails.status` → `complete`
- `User.onboardingCompleted` → `true`
- `User.plan.hasTuitionAccess` / `hasCandidateAccess` set per plan
- Clerk `publicMetadata.onboardingCompleted` → `true`

> **Note**: The onboarding UI shows a 30-day deletion countdown from `User.createdAt` for unpaid users. The `PATCH` endpoint sets a 72-hour `expiresAt` on `OnboardingDetails` — these timers serve related but distinct purposes.
