# Payments API

AOTF uses **Razorpay** exclusively for **provider registration/onboarding payments** (one-time fee). Consumer/guardian payments are handled offline and tracked manually in `PostLedger`.

## Fee Structure

| Component | Value |
|---|---|
| Base rate | 2% of transaction amount |
| GST on base | 18% of the 2% |
| **Effective rate** | **2.36% per transaction** |

Example: â‚ą500 registration fee â†’ Razorpay deducts â‚ą11.80 â†’ Provider receives â‚ą488.20

## Create Order

```http
POST /api/v1/payments/create-order
Authorization: Bearer <clerk_jwt>
Content-Type: application/json
Origin: https://aotf.in
```

Creates a Razorpay order for the provider registration fee. Only callable during onboarding (user must have `onboardingCompleted: false`).

**Request Body**:
```json
{ "plan": "teacher" }
```

Valid plans: `"teacher"`, `"teacher_candidate"`

**Response 200**:
```json
{
  "orderId": "order_ABC123",
  "amount": 50000,
  "currency": "INR",
  "keyId": "rzp_live_..."
}
```

`amount` is in **paise** (1/100 of a rupee). â‚ą500 = `50000` paise.

The client uses `orderId` and `keyId` to open the Razorpay modal:

```js
// Client-side
const razorpay = new window.Razorpay({
  key: keyId,
  order_id: orderId,
  amount: amount,
  currency: "INR",
  name: "AOTF",
  handler: async (response) => {
    // Call /api/v1/payments/verify with the response
    await fetch("/api/v1/payments/verify", {
      method: "POST",
      body: JSON.stringify(response),
    });
  },
});
razorpay.open();
```

## Verify Payment

```http
POST /api/v1/payments/verify
Authorization: Bearer <clerk_jwt>
Content-Type: application/json
Origin: https://aotf.in
```

Verifies the Razorpay HMAC signature and marks onboarding as complete.

**Request Body**:
```json
{
  "razorpay_order_id": "order_ABC123",
  "razorpay_payment_id": "pay_XYZ789",
  "razorpay_signature": "abc123..."
}
```

**Verification logic**:
```ts
const generatedSignature = crypto
  .createHmac("sha256", process.env.RAZORPAY_KEY_SECRET!)
  .update(`${orderId}|${paymentId}`)
  .digest("hex");

if (generatedSignature !== razorpay_signature) {
  return 400 // Invalid signature
}
```

**Response 200**:
```json
{ "success": true, "message": "Payment verified. Onboarding complete." }
```

**Side effects on success**:
1. Creates a `Payment` document in MongoDB
2. Updates `OnboardingDetails.status` â†’ `"complete"`
3. Sets `User.onboardingCompleted: true`
4. Updates `User.plan.hasTuitionAccess` / `hasCandidateAccess`
5. Updates `User.registrationPaymentId` with the `Payment._id`
6. Syncs Clerk `publicMetadata.onboardingCompleted: true`

**Response 400** (invalid signature):
```json
{ "error": "Payment verification failed. Invalid signature." }
```

## Razorpay Webhook

```http
POST /api/v1/webhooks/razorpay
```

Receives Razorpay payment events for server-side verification and reconciliation.

**Authentication**: Razorpay signs requests with `X-Razorpay-Signature` header using `RAZORPAY_WEBHOOK_SECRET`.

**Handled Events**:

| Event | Action |
|---|---|
| `payment.captured` | Confirm payment in DB, update status to `captured` |
| `payment.failed` | Update payment status to `failed`, log the failure reason |

**Why Both verify and Webhook?**

- `POST /api/v1/payments/verify` is called immediately by the client for fast UX feedback
- The webhook provides a **reliable server-to-server** confirmation for reconciliation
- If the client-side verify fails (network issue), the webhook still marks the payment correctly
