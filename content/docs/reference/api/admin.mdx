# Admin APIs

Admin endpoints require a valid Clerk session **and** the corresponding permission in `Admin.permissions`. The permission is checked against the MongoDB `Admin` document (JWT is the cache, DB is the source of truth).

## Base Paths

Admin APIs are spread across two base paths:

| Path | Purpose |
|---|---|
| `/api/v1/admin/*` | Application-level admin operations |
| `/api/admin/*` | Infrastructure-level admin operations (login, join) |

## Admin Login

```http
POST /api/admin/login
Content-Type: application/json
```

Admin sign-in is handled via the Clerk standard sign-in flow. This endpoint handles additional post-login checks:

- Verifies `isAdmin: true` in Clerk `publicMetadata`
- Checks `Admin.isActive` and `Admin.isLocked` in MongoDB
- Records login attempt in `LoginAttempt` collection
- After **5 failed attempts**: sets `Admin.isLocked: true`

## Admin Join (Invite-Based)

```http
POST /api/admin/join
Content-Type: application/json
```

Creates a new admin account using an invite token. **No auth required** — the token is the authenticator.

```json
{
  "token": "inv_abc123...",
  "password": "NewStrongPass123!"
}
```

**Flow**:
1. Verify token exists in `AdminInvite` collection and is not expired
2. Create Clerk user with the pre-configured role from the invite
3. Create `Admin` document in MongoDB
4. Mark invite as used
5. Return `{ success: true }`

Rate-limited to **5 requests per IP per minute**.

## Admin Management

### List Admins

```http
GET /api/v1/admin/admins
Authorization: Bearer <admin_clerk_jwt>
```

**Permission**: `canManageAdmins`

Returns all admin accounts with their roles, permissions, and activity status.

### Get Admin

```http
GET /api/v1/admin/admins/:adminId
Authorization: Bearer <admin_clerk_jwt>
```

### Create Admin Invite

```http
POST /api/v1/admin/admins/invite
Authorization: Bearer <admin_clerk_jwt>
Content-Type: application/json
Origin: https://aotf.in
```

**Permission**: `canCreateAdmins`

```json
{
  "email": "newadmin@aotf.in",
  "role": "support_admin",
  "name": "Support Staff"
}
```

Creates an `AdminInvite` document with a secure token. The invite link is emailed to the recipient.

### Update Admin

```http
PATCH /api/v1/admin/admins/:adminId
Authorization: Bearer <admin_clerk_jwt>
Content-Type: application/json
Origin: https://aotf.in
```

**Permission**: `canEditAdmins`

### Deactivate Admin

```http
DELETE /api/v1/admin/admins/:adminId
Authorization: Bearer <admin_clerk_jwt>
Origin: https://aotf.in
```

**Permission**: `canDeactivateAdmins`

Soft delete — sets `Admin.isActive: false`. Does not delete the Clerk account.

### Reset Admin Password

```http
POST /api/v1/admin/admins/:adminId/reset-password
Authorization: Bearer <admin_clerk_jwt>
Content-Type: application/json
Origin: https://aotf.in
```

**Permission**: `canResetAdminPasswords`

Generates a Clerk password reset link and sets `requirePasswordChange: true`.

## Analytics

```http
GET /api/v1/admin/analytics
Authorization: Bearer <admin_clerk_jwt>
```

**Permission**: `canViewAnalytics`

Returns aggregated stats:
- Total posts/jobs by status
- Application conversion rates (applied → approved)
- Monthly registration counts
- Enquiry resolution rate
- Top subjects/locations by demand

## Calendar

```http
GET /api/v1/admin/calendar
Authorization: Bearer <admin_clerk_jwt>
```

Query parameters:
- `from` — Start date (ISO-8601)
- `to` — End date (ISO-8601)
- `category` — Filter: `tuition`, `job`, `enquiry`, `feedback`, `reminder`
- `adminId` — Filter by assigned admin

Returns `CalendarEvent` documents for the specified range.

## Todo Events

```http
GET /api/v1/admin/todos
POST /api/v1/admin/todos
PATCH /api/v1/admin/todos/:todoId
DELETE /api/v1/admin/todos/:todoId
Authorization: Bearer <admin_clerk_jwt>
```

CRUD for admin todo/reminder events. All operations sync to `CalendarEvent` via Mongoose hooks.

**Todo categories**: `tuition`, `enquiry`, `job`, `feedback`

## Audit Logs

```http
GET /api/v1/admin/audit-logs
Authorization: Bearer <admin_clerk_jwt>
```

**Permission**: `canViewAuditLogs`

Query parameters:
- `action` — Filter by action type (28 types, see [Admin Tracking](/docs/explanations/admin-tracking))
- `adminId` — Filter by admin who performed the action
- `from`, `to` — Date range

Returns `AuditLog` documents for security-sensitive actions.

## Admin Activity Log

```http
GET /api/v1/admin/activity-logs
Authorization: Bearer <admin_clerk_jwt>
```

Returns `AdminActivityLog` documents for day-to-day CRM operations. See [Admin Tracking](/docs/explanations/admin-tracking) for the distinction between AuditLog and AdminActivityLog.

## Payments (Admin View)

```http
GET /api/v1/admin/payments
Authorization: Bearer <admin_clerk_jwt>
```

**Permission**: `canViewPayments`

List all provider registration payments with Razorpay IDs and amounts.

```http
POST /api/v1/admin/payments/refund
Authorization: Bearer <admin_clerk_jwt>
```

**Permission**: `canProcessRefunds`

Initiates a Razorpay refund for a given payment ID.
