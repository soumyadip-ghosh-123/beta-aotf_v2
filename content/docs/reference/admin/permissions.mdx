# Admin Permissions Matrix

AOTF uses a fine-grained permission system with **23 boolean flags** stored in the `Admin.permissions` object. Permissions are set at account creation (via `pnpm create:admin` or admin invite) and can be updated by a `super_admin` or `admin` with `canEditAdmins`.

## Role Summaries

### Super Admin (`super_admin`)
Full access to all 23 permissions. Can do everything including user blocking, refunds, analytics, data export, and full admin lifecycle management (create, reset password, terminate).

### Admin (`admin`)
Broad operational access. Can manage content (posts, jobs, applications), handle all CRM operations, view payments, manage other admins (except create, reset password, terminate). Cannot access analytics or export data.

### Support Admin (`support_admin`)
Limited to customer-facing CRM work only. Can handle enquiries, provide feedback, update enquiry status, and schedule calls with applicants. Cannot create content, access financial data, or manage admin accounts.

### CRM (`crm`)
Content creation + full CRM access, no financial or admin management. Designed for CRM operators who need to create posts/jobs from enquiries and handle the full enquiry-to-assignment workflow, but don't need financial visibility or admin control.

## How Permissions Are Checked

```ts
// In an API route handler:
const { sessionClaims } = await auth();
const meta = sessionClaims?.publicMetadata as AdminMetadata;

// Fast check: JWT cache (no DB call)
if (!meta?.permissions?.canManagePosts) {
  return NextResponse.json({ error: "Insufficient permissions" }, { status: 403 });
}

// For high-security operations: always cross-check DB
const admin = await Admin.findOne({ clerkId: userId }).lean();
if (!admin?.permissions?.canProcessRefunds) {
  return NextResponse.json({ error: "Insufficient permissions" }, { status: 403 });
}
```

The JWT metadata is checked first for performance. The DB is always the authority for financial and security operations.

## Updating Permissions

Permissions can be updated via:

1. **Admin Panel** → Admin Management → Edit Admin → Toggle permissions
2. **Re-running `pnpm create:admin`** with the same username (idempotent update)
3. **Direct MongoDB update** (emergency only, use carefully):
   ```js
   db.admins.updateOne(
     { username: "admin1" },
     { $set: { "permissions.canViewAnalytics": true } }
   )
   // Note: Also update Clerk publicMetadata for the change to reflect in JWT
   ```
