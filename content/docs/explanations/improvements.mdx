# Areas of Improvement

AOTF currently runs efficiently on free-tier infrastructure. However, as the platform scales, several architectural limitations will need to be addressed.

## 1. Rate Limiting Architecture

**Current State**: The rate limiter (`lib/rate-limit.ts`) uses an **in-memory sliding window**.
**The Problem**: In a serverless environment like Vercel, requests are routed to multiple isolated function instances. An in-memory cache means each instance has its own separate counter. If a user spams the endpoint, they might hit instance A 5 times, then instance B 5 times, effectively bypassing the limit.
**The Fix**: Implement a distributed rate limiter using **Upstash Redis** and `@upstash/ratelimit`. This ensures a single source of truth across all serverless instances.

## 2. Webhook Idempotency Guarantees

**Current State**: Webhook handlers (`/api/v1/webhooks/clerk`) use a `WebhookEvent` MongoDB collection. The handler attempts to `create()` a record with a unique index. If it fails with a duplicate key error, the event is ignored.
**The Problem**: If the Vercel function times out *after* creating the `WebhookEvent` but *before* finishing the actual work (e.g., creating the `User`), the webhook will be marked as processed, and Clerk's automatic retry will be ignored. The data will be out of sync.
**The Fix**: Implement a proper two-phase commit or use a transactional queue (like Upstash QStash) to decouple the webhook acknowledgement from the actual processing work.

## 3. Database Transactions

**Current State**: Operations that require modifying multiple collections (like creating a User and a Profile during onboarding) use standard sequential `await` calls.
**The Problem**: If the server crashes between creating the User and creating the Profile, the database is left in an inconsistent state. MongoDB Atlas M0 (Free Tier) does **not** support multi-document ACID transactions.
**The Fix**: Upgrade to a paid MongoDB Atlas cluster (M10+) which supports replica sets and transactions. Then, wrap critical multi-document writes in `session.withTransaction()`.

## 4. Search Functionality

**Current State**: The platform uses basic MongoDB text search (`$regex` or text indexes) for filtering posts, jobs, and users.
**The Problem**: As data grows, regex queries become slow. MongoDB's basic text search lacks typo tolerance and advanced ranking.
**The Fix**: Implement **MongoDB Atlas Search** (Lucene-based) or integrate a dedicated search engine like Algolia or Typesense for typo-tolerant, fast search.

## 5. File Upload Architecture

**Current State**: Avatar uploads are handled client-side directly to Cloudinary using an unsigned upload preset.
**The Problem**: Unsigned presets are less secure. Malicious users could theoretically spam the Cloudinary bucket.
**The Fix**: Move to **signed uploads**. The client requests an upload signature from a protected Next.js API route, and uses that signature to upload to Cloudinary.

## 6. Cold Start Mitigation

**Current State**: AOTF mitigates Vercel and MongoDB cold starts via `instrumentation.ts` warm-ups and robust retry logic in `dbConnect()`.
**The Problem**: The first user to hit the site after an hour of inactivity still experiences a 4–8 second load time.
**The Fix**: 
1. Upgrade MongoDB to M10 (removes auto-pause).
2. Set up a Vercel Cron Job to ping `/api/health` every 5 minutes to keep the serverless function warm.

## 7. Email Delivery

**Current State**: Emails are sent synchronously during API requests via Resend.
**The Problem**: If the Resend API is slow or down, the AOTF API request hangs or fails.
**The Fix**: Move email sending to a background job queue (e.g., Upstash QStash). The API request immediately returns success, and the email is sent asynchronously.
