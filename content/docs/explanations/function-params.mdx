# Vercel Function Parameters

Because AOTF is hosted on Vercel, it is bound by Vercel's serverless function constraints. Understanding these parameters is critical for writing robust API routes.

## The 10-Second Timeout

On the Vercel Hobby (Free) tier, every serverless function has a **maximum execution time of 10 seconds**.

If a function takes longer than 10 seconds to return a response, Vercel violently kills the function and returns a `504 Gateway Timeout` to the client.

### Implications for AOTF

1. **Database Queries**: If MongoDB is slow (e.g., during a cold start), the query *must* either complete or fail before the 10-second mark. `lib/db.ts` uses a `serverSelectionTimeoutMS` of 15 seconds, which technically exceeds the Vercel timeout. This means if MongoDB is completely down, Vercel will kill the function before Mongoose throws its own timeout error.
2. **Third-Party APIs**: Calls to Razorpay or Resend must be fast.
3. **Heavy Processing**: You cannot do heavy PDF generation, image manipulation, or large CSV exports synchronously in an API route. These operations must be offloaded or optimized.

## The 4.5MB Payload Limit

Vercel limits the size of request and response bodies:

- **Request Body**: Max 4.5 MB
- **Response Body**: Max 4.5 MB

### Implications for AOTF

1. **Avatar Uploads**: AOTF avoids the request limit by having the client upload images directly to Cloudinary. The API route only receives the Cloudinary URL (a small string).
2. **Data Exports**: If the admin team exports a massive list of users or ledgers, the JSON response cannot exceed 4.5 MB. If the database grows significantly, exports will need to be paginated or streamed via a different mechanism.

## Memory Limit

Hobby tier functions have a **1024 MB memory limit**.

For a standard CRUD Next.js app like AOTF, this is massive and rarely an issue. However, if a query uses `.find()` without `.lean()` on thousands of documents, Mongoose will instantiate heavy document objects and could blow out the memory.

**Rule**: Always use `.lean()` for read-only queries.

## Stateless Nature

Vercel functions spin up and down dynamically based on traffic.

### Implications for AOTF

1. **No In-Memory Storage**: You cannot store data in variables (like `let adminCache = []`) and expect it to persist across requests. Every request might hit a different serverless instance.
2. **The Rate Limiter Flaw**: As noted in [Areas of Improvement](/docs/explanations/improvements), the current in-memory rate limiter only limits traffic per *instance*, not globally across the entire app.

## Overcoming Limits

If the business outgrows these limits, the upgrade path is:
1. Upgrade to **Vercel Pro**: Increases timeout to 15 seconds (max 900s) and memory to 3008 MB.
2. Migrate off serverless: Move the backend to a long-running Node.js server (e.g., Railway, Render, DigitalOcean) where there are no arbitrary request timeouts.
