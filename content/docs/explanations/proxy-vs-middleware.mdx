# proxy.ts vs middleware.ts

## The Short Answer

In **Next.js 16+**, the `middleware.ts` file convention was replaced by `proxy.ts`. AOTF uses `proxy.ts` because the project runs on Next.js 16.1.

## What Changed in Next.js 16

Prior to Next.js 16, middleware was defined in `middleware.ts` (or `middleware.js`) at the project root. Starting with Next.js 16, Next.js introduced `proxy.ts` as the new convention for server-side middleware/routing logic.

> **Official documentation**: [Next.js proxy.ts API Reference](https://nextjs.org/docs/app/api-reference/file-conventions/proxy)

Key differences:

| Feature | `middleware.ts` (pre-v16) | `proxy.ts` (v16+) |
|---|---|---|
| File name | `middleware.ts` | `proxy.ts` |
| Config export | `export const config` | `export const config` (same) |
| Matcher | `config.matcher` array | `config.matcher` array (same) |
| Edge runtime | Default | Default |
| Clerk compat | `clerkMiddleware` | `clerkMiddleware` (same import path) |

The core logic is identical — the **filename** is the only structural change.

## The config Export IS the Matcher

`proxy.ts` exports **exactly two things**:

```ts
// proxy.ts
import { clerkMiddleware, createRouteMatcher } from "@clerk/nextjs/server";
import { NextResponse } from "next/server";

// 1. The middleware function (default or named export)
export default clerkMiddleware(async (auth, request) => {
  // ... route guard logic
});

// 2. The config — this IS the Next.js matcher configuration
export const config = {
  matcher: [
    // Match everything except static files and Next.js internals
    "/((?!_next|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)",
    // Always run for API routes
    "/(api|trpc)(.*)",
  ],
};
```

The `config.matcher` array tells Next.js **which paths** the proxy function runs on. It's a regex-based matcher — everything matching runs through `proxy.ts`, everything else serves files directly.

## What proxy.ts Does in AOTF

AOTF's `proxy.ts` handles all three concerns:

### 1. Webhook Pass-Through

Webhook endpoints must receive raw headers (Svix/Razorpay signature headers). `clerkMiddleware` strips these. The proxy detects webhook paths and bypasses middleware:

```ts
// proxy.ts (simplified)
if (pathname.startsWith("/api/v1/webhooks/")) {
  // Don't run Clerk middleware — pass through with original headers
  return NextResponse.next({ request: { headers: requestHeaders } });
}
```

### 2. Onboarding Gate

After authentication, the proxy checks if the user has completed onboarding:

```ts
const { userId, sessionClaims } = await auth();

if (userId && !isPublicRoute(request)) {
  const onboardingCompleted = 
    (sessionClaims?.publicMetadata as any)?.onboardingCompleted;
  
  if (!onboardingCompleted && !pathname.startsWith("/onboarding")) {
    return NextResponse.redirect(new URL("/onboarding", request.url));
  }
}
```

### 3. Admin Route Guard

Admin routes check for `isAdmin: true` in the JWT:

```ts
if (pathname.startsWith("/admin") || pathname.startsWith("/api/v1/admin")) {
  const meta = sessionClaims?.publicMetadata as any;
  
  if (!meta?.isAdmin) {
    return NextResponse.redirect(new URL("/sign-in", request.url));
  }
}
```

### 4. Public Route Definition

Some routes are public (no auth required):

```ts
const isPublicRoute = createRouteMatcher([
  "/",
  "/sign-in(.*)",
  "/sign-up(.*)",
  "/posts(.*)",
  "/jobs(.*)",
  "/u/(.*)",         // Provider public profiles
  "/api/v1/enquiry", // Public enquiry form (POST)
  "/api/health",
  "/api/openapi.json",
  "/api/search",
  "/docs(.*)",
]);
```

## Why Not Keep middleware.ts?

If you upgrade from an older Next.js project:

1. **Rename** `middleware.ts` → `proxy.ts`
2. **Exports stay the same** — the `config` export and default export work identically
3. **Imports stay the same** — Clerk SDK imports from the same paths

There is no code change needed inside the file — just the filename.

## Performance Characteristics

`proxy.ts` runs on the **Edge Runtime** by default — it executes at Vercel's edge network nodes, closer to users geographically. This means:

- JWT verification is fast (no round-trip to origin)
- DB queries are **not possible** in the proxy (no Node.js `net` module on Edge)
- All DB work happens in Route Handlers running in Node.js serverless functions

This is why the proxy uses `sessionClaims.publicMetadata` (JWT-embedded data) for auth checks instead of querying MongoDB.
