# The Admin Workflow

AOTF is designed to digitize a highly manual recruitment/matching business. The platform does not fully automate matching; instead, it provides a structured **CRM (Customer Relationship Management)** and **FRM (Freelancer Relationship Management)** toolkit for the admin team.

## Overview of Daily Operations

1. **Intake**: Consumers submit needs. Admins log these as Enquiries.
2. **Publishing**: Admins convert Enquiries to open Posts or Jobs.
3. **Filtering**: Providers apply. Admins review applications.
4. **Matching**: Admins schedule Demos (DC).
5. **Confirmation**: Consumers approve (GC). Admins mark the match.
6. **Billing**: Admins generate invoices and track payment.

## 1. Handling Enquiries

Consumers reach out via the website form, WhatsApp, or phone calls.

- If via the form: The enquiry appears in the `new` state in the Admin Dashboard.
- If via call/WhatsApp: The admin manually creates an Enquiry record.

The admin calls the consumer, discusses the requirement, and updates the status to `in_progress`.

## 2. Creating Posts/Jobs

Once the requirement is clear, the admin clicks **"Create Post"** directly from the Enquiry view.

- This links the `enquiryId` to the `Post`.
- The `Post` is now public on `aotf.in/posts`.
- The PostLedger (Tuitions Google Sheet) is automatically updated with a new row.

## 3. Reviewing Applications

Providers browse the public board and apply.

Admins view the "Applications" tab for a specific Post. They see:
- Provider's profile (experience, qualifications)
- Cover letter
- Admin notes (internal)

The admin filters down the list and selects 1–3 promising candidates.

## 4. The DC/GC Flow

### DC (Demo Call)
The admin contacts the shortlisted provider(s) and schedules a trial class/interview.
- The admin changes the application status to `DC`.
- A calendar event is created for the demo slot.

### GC (Guardian Confirmation)
After the demo, the admin contacts the consumer.
- If the consumer approves: Application status moves to `GC` (provisional approval).
- If the consumer rejects: Application status moves to `decline`.

## 5. Approval & Auto-Decline

Once everything is finalized, the admin clicks **Approve** on the winning application.

The system performs an atomic cascade:
1. The winning application becomes `approved`.
2. **ALL OTHER** active applications for this post become `auto_declined`.
3. The Post status becomes `matched`.
4. Notifications are sent to all involved providers.

This entirely removes the manual burden of sending rejection messages to 20+ applicants.

## 6. Ledger & Invoicing

With the teacher assigned, the admin manages the financial side via the `PostLedger` (which syncs to Google Sheets).

- The admin generates an **Invoice** (which creates a PDF-ready versioned document).
- When the consumer pays, the admin logs the payment amount and date.
- The admin tracks if the teacher has received their cut.

This completes the lifecycle of a single placement.
