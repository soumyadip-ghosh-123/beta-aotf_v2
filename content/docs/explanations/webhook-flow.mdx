# Webhook Flow

AOTF receives webhooks from **Clerk** (user lifecycle) and **Razorpay** (payment confirmation). Both are documented in detail in the [Webhooks API reference](/docs/reference/api/webhooks).

## Why Webhooks Matter

| Source | Primary role |
|---|---|
| Clerk `user.created` | Creates `User` + `Profile` for self-service sign-ups |
| Clerk `user.updated` / `user.deleted` | Sync profile changes and soft deletes |
| Razorpay `payment.captured` | Reconcile payments if client verify fails |

## Migration vs Webhook

For [legacy migration](/docs/how-to/migrate-legacy-users), migration scripts call `seedClerkUserInMongo()` **immediately** after each Clerk `createUser`. The `user.created` webhook is a **backup** if:

- The script's Mongo seed step failed
- The user signs in before the script finishes seeding

Both paths use the same seed logic in `lib/migration/seed-clerk-user.ts` (mirrored in `scripts/lib/seed-clerk-user.mjs`).

## proxy.ts Bypass

Webhook routes skip Clerk middleware so Svix signature headers are preserved:

```
/api/v1/webhooks/clerk
/api/v1/webhooks/razorpay
```

See [proxy.ts vs middleware](/docs/explanations/proxy-vs-middleware) for the full request routing story.

## Idempotency

Clerk events are deduplicated via the `WebhookEvent` collection (duplicate `svix-id` → HTTP 200, no re-processing). Failed processing deletes the event record so Clerk retries can succeed.

For the full event tables and handler steps, see [Webhooks API](/docs/reference/api/webhooks).
