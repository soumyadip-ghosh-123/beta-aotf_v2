# Admin Tracking & Audit

AOTF tracks admin actions across two distinct levels of granularity: **AuditLog** and **AdminActivityLog**.

## Why Two Logs?

| Feature | AuditLog | AdminActivityLog |
|---|---|---|
| **Purpose** | Security, compliance, high-risk actions | Day-to-day CRM operations |
| **Volume** | Low | High |
| **Examples** | Password reset, role change, data export | Status update, teacher assigned |
| **Data Kept** | Who, what, when, IP | Full `diff.before` and `diff.after` state |
| **Audience** | Super Admins | All Admins (to see who did what) |

---

## 1. AuditLog

**Collection**: `audit_logs`

The AuditLog is strictly for security and compliance. Only **Super Admins** and Admins with `canViewAuditLogs` can see these.

### The 28 Action Types

1. **User Management**: `user_blocked`, `user_unblocked`, `user_deleted`, `user_restored`
2. **Admin Management**: `admin_created`, `admin_updated`, `admin_deactivated`, `admin_reactivated`, `admin_terminated`, `admin_password_reset`, `admin_role_changed`, `admin_permissions_updated`, `admin_locked`, `admin_unlocked`
3. **Financial**: `refund_initiated`, `refund_approved`, `refund_rejected`
4. **Data Operations**: `data_exported`, `bulk_action_performed`
5. **Auth Events**: `login_success`, `login_failed`, `logout`, `password_changed`
6. **Content**: `post_deleted`, `job_deleted`
7. **Audit**: `audit_log_viewed`, `audit_log_exported`, `settings_changed`

### Logging Example

```ts
import { logAudit } from "@/lib/services/audit.service";

// When an admin changes another admin's role
await logAudit({
  action: "admin_role_changed",
  performedByAdminClerkId: currentUserId,
  performedByAdminName: currentUser.name,
  targetType: "Admin",
  targetId: targetAdmin._id.toString(),
  ipAddress: req.ip,
  userAgent: req.headers.get("user-agent"),
  metadata: {
    fromRole: oldRole,
    toRole: newRole
  }
});
```

## 2. AdminActivityLog

**Collection**: `admin_activity_logs`

The Activity Log tracks operational changes. If a post's location is edited, or an enquiry is marked as `resolved`, it goes here. This enables the UI to show "Last edited by X at Y" and provides an undo trail.

### The 7 Modules

| Module | Purpose |
|---|---|
| `CRM` | Enquiries, status updates |
| `FRM` | Posts, jobs, applications |
| `COMMS` | Emails sent, calls logged |
| `CALENDAR` | Events added, edited |
| `LEDGER` | PostLedger (tuition) updates |
| `ADMIN_MGMT` | Operational admin updates |
| `USER_MGMT` | Operational provider updates |

### Diff Tracking

The Activity Log requires you to provide the state before and after the change:

```ts
import { logActivity } from "@/lib/services/activity.service";

// Save the state before the update
const stateBefore = enquiry.toObject();

// Perform the update
enquiry.currentStatus = "in_progress";
await enquiry.save();

// Log the change
await logActivity({
  module: "CRM",
  action: "updated_enquiry_status",
  adminClerkId: admin.clerkId,
  adminName: admin.name,
  entityId: enquiry._id.toString(),
  entityType: "Enquiry",
  diff: {
    before: stateBefore,
    after: enquiry.toObject()
  },
  ipAddress: req.ip,
  browser: browserInfo,
  os: osInfo
});
```

## Admin Login Tracking

**Collection**: `login_attempts`

Every admin login attempt (success or failure) is logged.

- Tracks `username`, `ipAddress`, `userAgent`, and `success` boolean.
- After **5 consecutive failed attempts** for a given username, the `Admin` account is locked (`isLocked: true`).
- A `super_admin` must unlock the account via the admin panel.
