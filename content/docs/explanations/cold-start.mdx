# Cold Starts & Free Tier Resilience

AOTF runs on free-tier infrastructure, which introduces specific latency characteristics that the codebase is explicitly designed to handle.

## The Two Cold Start Problems

### 1. MongoDB Atlas M0 Auto-Pause

MongoDB Atlas M0 (free tier) **automatically pauses the cluster after 60 minutes of inactivity**. When a request comes in after a period of inactivity:

- Atlas must "resume" the cluster (spin up processes, load data into memory)
- This takes **4–10 seconds** typically
- The first query must wait while Atlas resumes

### 2. Vercel Serverless Cold Start

Vercel's serverless functions are created fresh for each request after a period of inactivity. A cold serverless function:

- Takes **200–800ms** to initialise the Node.js runtime
- Must re-establish the MongoDB connection (even though the cached connection is in the module scope, the module itself is loaded fresh)

## Combined Worst Case

In the absolute worst case (cluster paused + function cold):

```
Vercel function init:     ~500ms
MongoDB cluster wake:     ~8,000ms
Initial connection:       ~500ms
First query:             ~100ms
─────────────────────────────────
Total first request:     ~9,100ms
```

This is near the Vercel free tier's **10-second function timeout**. The codebase handles this carefully.

## The dbConnect() Retry Strategy

```ts
// lib/db.ts
const MAX_RETRIES = 3;
const BASE_DELAY_MS = 2000;

export default async function dbConnect() {
  // Return cached connection if available
  if (cached.conn) return cached.conn;

  for (let attempt = 1; attempt <= MAX_RETRIES; attempt++) {
    try {
      const conn = await mongoose.connect(MONGODB_URI, {
        serverSelectionTimeoutMS: 15_000,  // Give Atlas 15s to wake
        socketTimeoutMS: 45_000,
      });
      cached.conn = conn;
      return conn;
    } catch (err) {
      if (attempt === MAX_RETRIES) throw err;
      
      const delay = BASE_DELAY_MS * Math.pow(2, attempt - 1); // 2s → 4s → 8s
      console.warn(`[dbConnect] Attempt ${attempt} failed, retrying in ${delay}ms`);
      await new Promise(r => setTimeout(r, delay));
    }
  }
}
```

| Attempt | Delay Before | Cumulative |
|---|---|---|
| 1 | 0ms (immediate) | 0ms |
| 2 | 2,000ms | 2,000ms |
| 3 | 4,000ms | 6,000ms |

With `serverSelectionTimeoutMS: 15_000`, each attempt gives Atlas 15 seconds to respond. The total wait before giving up is approximately **6s delay + 15s selection timeout = 21 seconds** — but in practice, Atlas wakes in 4–8s so the first or second attempt usually succeeds.

## instrumentation.ts — Startup Warm-Up

Next.js `instrumentation.ts` runs on server startup (before any requests are handled). AOTF uses it to pre-warm the MongoDB connection:

```ts
// instrumentation.ts
export async function register() {
  if (process.env.NEXT_RUNTIME === "nodejs") {
    const { default: dbConnect } = await import("./lib/db");
    
    // Fire-and-forget warm-up — don't block startup
    dbConnect()
      .then(() => console.log("[instrumentation] MongoDB pre-warmed"))
      .catch(err => console.warn("[instrumentation] Pre-warm failed:", err));
  }
}
```

**Why it helps**: When Vercel deploys a new function instance, `instrumentation.ts` runs immediately. By the time the first real user request arrives, the connection is already established and cached.

**Limitation**: On Vercel's free tier, function instances are spun up per-request. `instrumentation.ts` helps when the function is already warm from a recent request, but a truly cold function re-runs it.

## DNS Override for Development

On some development setups, DNS resolution for Atlas cluster hostnames fails:

```ts
// lib/db.ts — development DNS override
import dns from "dns";

if (process.env.NODE_ENV !== "production") {
  dns.setServers(["1.1.1.1"]); // Cloudflare DNS
}
```

This resolves Atlas hostnames reliably without relying on the developer's network DNS.

## Health Endpoint Behavior on Cold Start

`GET /api/health` specifically uses `dbConnect()` — so the first health check after a cold period will be slow (4–10s) but will return `{ "status": "ok" }` once Atlas wakes. This is intentional: uptime monitors hitting `/api/health` every minute keep Atlas from auto-pausing in production.

## Recommendations

1. **Set up an uptime monitor** — ping `/api/health` every 5 minutes to prevent Atlas auto-pause
2. **First request warning** — warn users in your admin UI that the first load after inactivity may be slow
3. **Upgrade when needed** — if cold starts become critical, upgrade to MongoDB Atlas M10 (paid, no auto-pause) and Vercel Pro (longer function timeout)
