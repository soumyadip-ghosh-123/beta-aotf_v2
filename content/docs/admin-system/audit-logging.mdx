## Overview

Every significant admin action is recorded in the `AuditLog` collection, creating a comprehensive trail of who did what, when, and from where.

---

## What Gets Logged

| Action | Trigger |
|--------|---------|
| `admin.created` | New admin account created |
| `admin.updated` | Admin permissions or details changed |
| `admin.deactivated` | Admin account deactivated |
| `admin.reactivated` | Admin account reactivated |
| `admin.terminated` | Admin account terminated |
| `admin.password_reset` | Admin password reset by Super Admin |
| `admin.unlocked` | Locked admin account unlocked |
| `user.blocked` | User account blocked |
| `user.unblocked` | User account unblocked |
| `post.created` | Tuition post created |
| `post.updated` | Tuition post updated |
| `job.created` | Job listing created |
| `job.updated` | Job listing updated |
| `application.status_changed` | Application status updated |
| `enquiry.status_changed` | Enquiry status updated |

---

## Audit Log Model

```typescript
{
  adminId: ObjectId,          // Admin who performed the action
  adminClerkId: string,       // Clerk ID of the acting admin
  adminUsername: string,       // Username for display
  action: string,             // Action type (e.g., "admin.created")
  targetType: string,         // "Admin", "User", "Post", etc.
  targetId: string,           // ID of the affected resource
  targetIdentifier: string,   // Human-readable identifier
  details: object,            // Action-specific metadata
  changes: object,            // Before/after snapshot for updates
  ipAddress: string,          // Client IP address
  userAgent: string,          // Browser/client info
  createdAt: Date,            // Timestamp
}
```

---

## Change Tracking

For update operations, the `changes` field records a before/after snapshot:

```json
{
  "changes": {
    "before": {
      "role": "moderator",
      "permissions": {
        "canManagePosts": false
      }
    },
    "after": {
      "role": "admin",
      "permissions": {
        "canManagePosts": true
      }
    }
  }
}
```

## Accessing Audit Logs

### API Endpoint

```
GET /api/v1/admin/audit-logs
```

**Available to:** Super Admin and Sub-Superadmin

**Query Parameters:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `page` | `number` | Page number |
| `limit` | `number` | Results per page |
| `action` | `string` | Filter by action type |
| `adminId` | `string` | Filter by acting admin |
| `targetType` | `string` | Filter by target type |
| `startDate` | `string` | Filter from date (ISO 8601) |
| `endDate` | `string` | Filter to date (ISO 8601) |

### Admin UI

Audit logs can be viewed in the admin dashboard at `/admin/activity`.

## IP Address Tracking

The client IP is extracted from proxy headers:

```typescript
function getClientIp(request: NextRequest): string {
  return (
    request.headers.get("x-forwarded-for")?.split(",")[0]?.trim() ??
    request.headers.get("x-real-ip") ??
    "unknown"
  );
}
```

This works behind Vercel's CDN and most reverse proxies.

---

## Retention

Audit logs are retained indefinitely (no TTL index). For compliance and accountability, it's recommended to:

- Regularly back up the `auditlogs` collection
- Consider archiving old logs to cold storage for cost optimization
- Monitor collection size to ensure it doesn't impact query performance
